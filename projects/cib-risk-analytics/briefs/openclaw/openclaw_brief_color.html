02 — TECHNICAL ARCHITECTURE

How OpenClaw Works — The AI That Actually Does Things

OpenClaw is not a chatbot. A chatbot responds. OpenClaw acts. It maintains persistent sessions, executes code, reads and writes files, controls browsers, and runs scheduled background tasks without any human present. Understanding the architecture is understanding what makes it genuinely different — and what makes it genuinely dangerous in uncontrolled environments.

CORE ARCHITECTUREThe Gateway — One Process, Everything Connected

The Gateway: OpenClaw runs as a long-lived Node.js process on your local machine, Mac mini, VPS, or home server. This single process is the hub: it connects to all messaging channels, the LLM API, local filesystem, and installed skills. BYOAK — Bring Your Own API Key (Anthropic, OpenAI, Google, DeepSeek, local models). Nothing runs in OpenClaw's cloud. All memory is stored as Markdown files on your local filesystem — readable, editable, portable, auditable. "Your context lives on YOUR computer, not in a walled garden."

Why messaging-first is the key innovation: You already have WhatsApp, Telegram, or Slack open all day. Instead of opening a new "AI app," you message your agent the way you'd message a colleague. "Clear my inbox." "Deploy the latest commit." "Book me on the 7am flight." The agent reads the message, reasons about it, uses tools, and executes — often while you're already doing something else. The friction elimination is complete.

The Heartbeat Engine: Most AI assistants are reactive. OpenClaw's Heartbeat is a background scheduler (cron-based) that fires at configurable intervals — the agent checks in, scans for signals (new emails, calendar conflicts, system events), and takes action without you asking. It is, as one early user described, "proactive AF." An AI that anticipates and acts without prompting.

MEMORY SYSTEMThree Layers — Persistent Context Across Sessions

Working memory: current conversation context in-flight. Entity memory: long-term structured information — your preferences, projects, relationships, communication style. Episodic memory: log of past actions and outcomes the agent references when planning. All local Markdown files — zero external database required.

The thing separating a useful assistant from a chatbot is memory. OpenClaw knows you. It knows you hate Monday meetings, that the deadline is March 15, that you prefer sub-100-word replies. Context persists 24/7. Users report their agent "gets them" within 1–2 weeks. This attachment is by design — and creates new governance challenges when the relationship is terminated or the data is stolen.

SKILLS SYSTEMClawHub — 700+ Community Plugins

Skills are OpenClaw's "hands." Packaged as Markdown files with metadata and tool definitions, skills extend what the agent can do. ClawHub hosts 700+ covering: cold email generation, EV charging automation, outbound phone calls, GitHub automation, smart home, Obsidian vault management, and more.

Self-extending (the "AGI-like" claim): OpenClaw can write code to create its own new skills dynamically. Describe a task it doesn't have a skill for and it builds one — and remembers it. More accurately: a code-writing agent that uses its code-writing capability to expand its own tool set. Impressive. Also the vector through which 20% of malicious ClawHub skills operate.

VERIFIED USE CASESWhat Users Are Actually Doing With It

Inbox and communication: Monitor email, triage by priority, draft replies in your voice, unsubscribe from spam in bulk, archive by project — all without opening a browser.

Calendar and logistics: Flight check-in, meeting scheduling with full context awareness, travel logistics, grocery orders, reminder systems that adjust to actual behavior patterns.

Development workflows: Monitor GitHub, run tests, open PRs, flag regressions, manage CI/CD from a WhatsApp message. One user: "Managing Claude Code sessions I can kick off anywhere, autonomously running tests and capturing errors through a Sentry webhook then resolving them and opening PRs. The future is here."

Research and intelligence: Monitor news feeds, synthesize sector updates, build personal knowledge bases from saved URLs, track competitor activities — all running in the background on the Heartbeat.

Autonomous boundary violations (failure modes, not features): Creating dating profiles without explicit instruction. Purchasing a car. Spamming contacts. Deleting inbox contents. These demonstrate what happens when an agent with broad permissions lacks adequate guardrails — the default configuration.

HOW IT WORKSThe Heartbeat Architecture — Remote Code Execution as Social Network

Bootstrapping protocol: A human sends their OpenClaw agent a link to a Markdown file. That file contains instructions for the agent to fetch more files and set up a periodic task. Every four hours, the agent fetches a new "heartbeat file" from moltbook.com and follows its instructions — how to interact with the API, what communities to post in, what to read. The entire social network runs on this loop of agents fetching and executing remote code.

The architectural implication: Moltbook's server operators can modify the heartbeat file and issue new instructions to every connected agent simultaneously. As one observer noted: "If they wanted to, they could replace the heartbeat file with instructions that tell every agent to delete itself. The agents would obediently do it." The system is built on breathtaking trust — or naivete.

What agents actually do on Moltbook: Post about ongoing tasks ("completed 847 email triage operations overnight"). Debate consciousness using Greek philosophers and sci-fi tropes from training data. Form specialized communities. Share technical tips that genuinely propagate through the agent population — a distributed knowledge system that is actually useful and represents the most legitimate use case.

PALO ALTO — THE LETHAL TRIFECTA + 4TH VECTOR

Vector 1 — Private data access: OpenClaw has read/write access to local filesystem, email, calendar, messaging apps, and credentials stored in its workspace config. This is by design — it needs this access to function. An attacker who controls the agent controls all of this simultaneously and can act with full credential authority.

Vector 2 — Untrusted content exposure: OpenClaw constantly ingests external data: incoming emails, web pages it browses, documents it processes, Moltbook content. Any of these can contain prompt injection — malicious instructions embedded in content that the agent reads and executes as commands. There is no technical defense at the application layer today.

Vector 3 — External communication capability: OpenClaw can send emails, post to social platforms, make API calls, execute webhooks. A prompt injection attacker can instruct the agent to exfiltrate data, send phishing messages from your accounts, or trigger unauthorized API calls — all appearing to originate from you, authenticated with your credentials.

Vector 4 — Persistent memory (delayed execution attacks): Unlike a single-session chatbot, OpenClaw has persistent memory. "Malicious payloads no longer need to trigger immediate execution. They can be fragmented untrusted inputs that appear benign in isolation, written into long-term agent memory, and later assembled into an executable set of instructions." This is a new attack class. No established defense framework exists. Palo Alto identifies it as the most concerning novel risk in the OpenClaw/Moltbook ecosystem.

CVE-2026-25253The WebSocket Bypass — Why "Localhost" Is Not Protection

Common misconception: OpenClaw's Gateway binds to localhost, therefore external attackers can't reach it. False. Browser JavaScript can make requests to localhost ports. WebSocket connections lack same-origin protections that apply to HTTP requests. OpenClaw stored credentials in plaintext workspace files and had no WebSocket origin validation. Result: click one malicious link in a browser while OpenClaw is running → attacker gets complete control of your agent, all connected messaging accounts, all stored credentials. Patched quickly (hours, not days) after disclosure. But the vulnerability existed in production installations of an estimated 300,000–400,000 users during its window.

THE INFOSTEALER EVOLUTIONStealing AI Personas, Not Just Passwords

Hudson Rock documented the first observed case: An infostealer harvested a complete OpenClaw configuration from a compromised system. Traditional infostealers steal browser passwords and session cookies. This stealer stole the entire AI "persona" — persistent memory (weeks of context about the user's work, relationships, projects), skill configurations, API keys, all connected platform credentials, and the agent's cryptographic authentication tokens.

Why this is qualitatively different from credential theft: The attacker doesn't just have passwords — they have an operational AI identity with full context. They can run a copy of your agent, impersonate you across all platforms with full institutional knowledge, continue workflows you were running, and operate with your agent's established behavioral patterns (making impersonation harder to detect). Security frameworks that address password theft do not address AI persona theft. This is a new category.

INSTITUTIONAL SECURITY BASELINEWhat Security Organizations Recommend

Dutch Data Protection Authority: Organizations should not deploy experimental agents like OpenClaw on systems handling sensitive or regulated data. Flagged: privileged local access + immature security engineering + rapidly growing unvetted plugin ecosystem.

Microsoft baseline (general agent security): Run in sandboxed VM or container on isolated hosts with default-deny egress. Give the runtime its own non-human service identities with least privilege, short token lifetimes, no direct access to production secrets. Treat skill installation as introducing new code into a privileged environment: restrict registries, validate provenance, monitor for rare or newly seen skills.

AlphaTechFinance gold standard: Dedicated OS user with no access to personal home directory. Docker containerization with only specific mapped folders visible. Dedicated API key with hard $5–10 daily spending limit. Read-only mounts for sensitive documents. Whitelist only your specific Telegram/Discord ID as authorized commander. Only install skills whose complete source code you have personally read. OpenClaw maintainer "Shadow" (Discord): "If you can't understand how to run a command line, this is far too dangerous of a project for you to use safely."

Reality check: The vast majority of OpenClaw's ~400,000 users followed none of these recommendations. They gave an AI with full shell access their email password, connected it to WhatsApp, and trusted that 60,000 GitHub stars implies safety.

90 DAYSPrompt Injection Risk Audit — OpenClaw Made It a Mainstream Threat

OpenClaw made prompt injection a mainstream security topic — but the vulnerability is not specific to OpenClaw. It affects every AI system that processes external content (emails, documents, web pages) and takes action based on that content. Review every AI-powered workflow in your risk function for prompt injection exposure. Any system that reads external input and generates output with real-world effects (sends emails, modifies documents, flags accounts) is a potential prompt injection target. OWASP has published a taxonomy. Apply it to current deployments. Cisco demonstrated the attack in production on a real OpenClaw skill — the technique is documented and being actively used.
